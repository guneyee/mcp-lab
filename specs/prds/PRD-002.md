# PRD-002: Password Reset

## Problem Statement
Users who forget their passwords need a secure and reliable self-service flow to recover account access without support intervention.

## Target Users
- Registered users who cannot access their account
- Security and support teams monitoring account recovery events

## Functional Requirements

### FR-1: Reset Request
- Users can request password reset from login page
- System accepts email input and always returns a neutral success message
- Requests are rate-limited per email and IP

### FR-2: Secure Reset Token
- Reset links include signed, single-use tokens
- Tokens expire after 15 minutes
- Expired or reused tokens are rejected

### FR-3: Password Update
- Users can set a new password via valid reset link
- Password policy is enforced before submit
- Successful reset invalidates all active sessions except current flow

### FR-4: User Notification
- System sends confirmation email after password change
- Email includes timestamp, approximate location, and support guidance

### FR-5: Reset Session Security
- All existing sessions are revoked immediately after successful password reset
- User is prompted to re-authenticate on all active devices
- Security event is recorded with device metadata and timestamp

## Non-Functional Requirements

### NFR-1: Security
- OWASP ASVS aligned reset flow
- No account enumeration through API or UI messages
- Audit log for each reset lifecycle event

### NFR-2: Performance
- Reset request response time < 250ms (p95)
- Reset token validation < 100ms (p95)

### NFR-3: Reliability
- 99.9% availability for reset endpoints
- Email delivery retry policy with dead-letter handling

## Success Metrics
- 90% successful self-service reset completion rate
- < 1% reset abuse incidents per month
- < 5% support tickets related to password reset failures

## Out of Scope
- Social login recovery
- Multi-factor recovery fallback flow
- Admin-assisted reset workflows
